# Architecture

## Problem Statement

AI coding agents suffer from three core problems:
1. No persistence between sessions — agents forget everything
2. High token usage — redundant context loading, stale tool definitions, re-reading files
3. Context quality degrades as conversations grow — compaction loses important details

## Design Principles

- **MCP as protocol** — open standard, not proprietary feature
- **SQLite + FTS5 as storage** — single file, zero infrastructure, no embedding dependency at L0
- **Live query over pre-built index** — source files are source of truth
- **Git as change signal** — diffs over file watchers
- **Token ROI as primary metric** — every injected memory must earn its tokens
- **Progressive complexity** — ship simple, add layers when usage data demands it
- **Progressive disclosure** — never dump full memories; compact results -> detail on demand (from Engram)
- **Steal before building** — adopt proven patterns from Engram, SimpleMem; build only differentiators

## Unified Memory Model

One MCP server. One storage layer. Memory type drives behavior via `source_type`:

```
Memory node: {
  key:            string,          // semantic identifier (= Engram's topic_key)
  value:          any,             // the actual knowledge
  scope:          string,          // DAG path: /org/project/branch
  source_type:    enum,            // codebase | explicit | observed | derived
  source_ref:     string?,         // file:lines for codebase, null for others
  source_commit:  string?,         // git commit hash for codebase-derived
  confidence:     float,           // 1.0 explicit, 0.8 observed, decays
  revision_count: int,             // how many times updated (from Engram)
  duplicate_count: int,            // how many times seen (from Engram)
  session_id:     string?,         // which session created this
  created:        timestamp,
  accessed:       timestamp,
  last_seen_at:   timestamp        // for dedup window (from Engram)
}
```

### Source Types

| Type | Origin | Invalidation | Confidence |
|-|-|-|-|
| `codebase` | Parsed from source files | Git diff against source_ref | 1.0 (verified) |
| `explicit` | User said "remember X" | Never auto-invalidated | 1.0 (authoritative) |
| `observed` | Inferred from repeated behavior | Decays without reconfirmation | 0.5-0.9 (earned) |
| `derived` | Computed from other memories | Invalidate when parent changes | Inherits parent |
| `procedural` | Generated by skill completion | No decay (high-value patterns) | 0.9 (skill-verified) |

## Protocol Levels

### Level 0 — Core Memory (ship first, ~1200 lines)

```
memory/save       {key, value, scope?, source_type?, tags?}     // upsert by key+scope
memory/search     {query, scope?, type?, limit?}                // FTS5 search
memory/detail     {id}                                          // full untruncated content
memory/delete     {key, scope?}
memory/list       {scope?, source_type?, limit?}
memory/context    {scope?, limit?}                              // recent memories for injection
memory/session_start  {project, directory?}
memory/session_end    {summary?}
```

**Search is FTS5, not embeddings.** For coding agent memories (function names, error messages, config keys), keyword matching is sufficient and removes the embedding model dependency. Engram proved this works.

**Progressive disclosure.** `memory/search` returns compact results (~100 tokens each) with IDs. `memory/detail` returns full content. Agent drills in only when needed.

**Scope hierarchy.** DAG: `/` -> `/org/acme` -> `/project/api` -> `/branch/feature-x`. Reads walk up the scope chain (nearest match wins). Implementation: recursive CTE in SQLite.

**Deduplication.** SHA256 of normalized content + scope + key. Within rolling window, duplicates increment `duplicate_count` and update `last_seen_at`. Outside window, creates new entry.

**Privacy.** Strip `<private>...</private>` tags before storage, replace with `[REDACTED]`.

**Temporal anchoring.** Resolve relative timestamps ("yesterday", "last week") to ISO-8601 at write time (from SimpleMem).

### Level 1 — Provenance + Relations (~300 lines)

```
memory/relate     {source_id, target_id, relation}
memory/provenance {id} -> source chain
memory/stale      {scope?} -> memories needing revalidation
memory/conflicts  {scope?} -> contradicting memories
```

Relations: `derived_from`, `supersedes`, `conflicts_with`, `scoped_to`

Git-diff invalidation activates here: `codebase` memories check source_ref against current HEAD.

### Level 2 — Autonomous Behaviors (~500 lines, guided by usage data)

```
memory/extract     {source: "config" | "behavior" | "codebase"}
memory/consolidate {scope, strategy}
memory/budget      {max_tokens, scope} -> ranked memories fitting token budget
memory/metrics     {} -> hit rate, ROI, staleness stats
```

Adopt SimpleMem techniques:
- Entropy-aware filtering at write time (discard low-information content)
- Recursive consolidation (merge similar memories, similarity > 0.85)
- Adaptive retrieval depth (simple=3, complex=20 results)
- Optional embedding search alongside FTS5

## Cold Start

### 1. Config Extraction (first run)

Parse existing project files and populate Level 0 memories:

```
CLAUDE.md "always use bun"
  -> {key: "preference/package_manager", value: "bun", source_type: "explicit"}

package.json scripts.test = "vitest run"
  -> {key: "commands/test", value: "vitest run", source_type: "codebase", source_ref: "package.json"}

tsconfig.json compilerOptions
  -> {key: "config/typescript", value: {strict: true, ...}, source_type: "codebase"}

.env / .env.example
  -> {key: "config/env_vars", value: [list of var names], source_type: "codebase"}
```

### 2. Behavioral Observation (builds over time)

Track repeated search-then-use patterns across sessions:

```
Session 1: model runs `bun test`
Session 2: model searches for test command, runs `bun test`
Session 3: model searches again
  -> Auto-create: {key: "commands/test", value: "bun test", source_type: "observed", confidence: 0.9}
```

Signal: if the model looks something up more than twice, it should be a memory.

### 3. Explicit Storage

User says "remember X" or model decides to store. Least scalable, most reliable.

## Change Detection & Cache Strategy

### Committed State — Git Diff

```
Cache entry: {
  key:          "tools/Edit/schema",
  sourceFile:   "src/tools/edit.ts",
  sourceLines:  [15, 45],
  commitHash:   "abc123"
}

On query:
  if entry.commitHash == HEAD -> cache hit
  diff = git diff entry.commitHash..HEAD -- entry.sourceFile
  if diff doesn't touch lines 15-45 -> cache hit
  else -> invalidate, re-query from source
```

### Working Tree — Volatile Layer

fs.watch only on working tree (uncommitted edits during active development):

```
committed cache:  stable, invalidated by commits
working cache:    volatile, invalidated by file save
```

### Async Event Processing

Separated from cache invalidation. Runs at session start:

```
Git hook (post-commit)
  |-- Diff -> cache invalidation (mechanical, instant)
  |-- Commit + changed files -> event queue (processed async)

Event processor:
  - Reads queued changes
  - Classifies: breaking / rename / additive / cosmetic
  - Updates affected memory nodes
```

## Token Efficiency

### Progressive Disclosure (from Engram)

Never inject full memories into context. Three-layer pattern:

1. `memory/context` returns compact summaries (~100 tokens each, max 10-20 memories)
2. `memory/search` returns ranked results with IDs
3. `memory/detail` returns full content only when agent needs it

This alone can 5-10x token efficiency vs. naive memory injection.

### Token ROI Tracking

Every memory injection is measured:

```
Token ROI = tokens_saved / tokens_injected

Good: "test command is bun test" (10 injected, 200 saved = 20x)
Bad:  "project uses TypeScript strict mode..." (50 injected, 50 saved = 1x)
```

Track hit rate: what % of injected memories were referenced in the model's response.

### Where Tokens Actually Go

| Source | % of tokens | Solution |
|-|-|-|
| System prompts + tool definitions | ~30-40% | Dynamic tool loading |
| Conversation history pre-compaction | ~30-40% | Smarter compaction |
| Re-reading files already seen | ~10-15% | Read cache |
| Memory injection | ~5-10% | This system |

## Storage Schema

```sql
-- Schema versioning: uses SQLite user_version pragma via rusqlite_migration (AD-3)
-- No _schema_version table needed.

-- Core tables
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  project TEXT,
  directory TEXT,
  started_at TEXT NOT NULL,
  ended_at TEXT,
  summary TEXT,
  status TEXT DEFAULT 'active'
);

CREATE TABLE memories (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  key TEXT NOT NULL,                    -- semantic identifier (topic_key)
  value TEXT NOT NULL,                  -- the actual knowledge
  scope TEXT NOT NULL DEFAULT '/',      -- DAG path
  source_type TEXT NOT NULL DEFAULT 'explicit',  -- codebase|explicit|observed|derived|procedural
  source_ref TEXT,                      -- file:lines for codebase
  source_commit TEXT,                   -- git commit for codebase
  confidence REAL NOT NULL DEFAULT 1.0 CHECK (confidence >= 0.0 AND confidence <= 1.0),
  session_id TEXT REFERENCES sessions(id),
  tags TEXT,                            -- JSON array
  revision_count INTEGER DEFAULT 0,
  duplicate_count INTEGER DEFAULT 0,
  normalized_hash TEXT,                 -- SHA256 for dedup
  created_at TEXT NOT NULL,
  accessed_at TEXT NOT NULL,
  last_seen_at TEXT NOT NULL,
  deleted_at TEXT                       -- soft delete
);

-- FTS5 for text search (no embedding dependency)
CREATE VIRTUAL TABLE memories_fts USING fts5(
  key, value, tags, source_type, scope,
  content=memories, content_rowid=id
);

-- Auto-sync triggers
CREATE TRIGGER memories_ai AFTER INSERT ON memories BEGIN
  INSERT INTO memories_fts(rowid, key, value, tags, source_type, scope)
  VALUES (new.id, new.key, new.value, new.tags, new.source_type, new.scope);
END;

-- Normal updates (not soft-deleting) — AD-4
CREATE TRIGGER memories_fts_update AFTER UPDATE ON memories
WHEN new.deleted_at IS NULL BEGIN
  INSERT INTO memories_fts(memories_fts, rowid, key, value, tags, source_type, scope)
  VALUES ('delete', old.id, old.key, old.value, old.tags, old.source_type, old.scope);
  INSERT INTO memories_fts(rowid, key, value, tags, source_type, scope)
  VALUES (new.id, new.key, new.value, new.tags, new.source_type, new.scope);
END;

-- Soft-delete: remove from FTS5, don't re-insert
CREATE TRIGGER memories_fts_softdelete AFTER UPDATE ON memories
WHEN new.deleted_at IS NOT NULL AND old.deleted_at IS NULL BEGIN
  INSERT INTO memories_fts(memories_fts, rowid, key, value, tags, source_type, scope)
  VALUES ('delete', old.id, old.key, old.value, old.tags, old.source_type, old.scope);
END;

CREATE TRIGGER memories_fts_delete AFTER DELETE ON memories BEGIN
  INSERT INTO memories_fts(memories_fts, rowid, key, value, tags, source_type, scope)
  VALUES ('delete', old.id, old.key, old.value, old.tags, old.source_type, old.scope);
END;

-- Runtime metadata (AD-19)
CREATE TABLE IF NOT EXISTS _metadata (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Stores: db_created_at, last_vacuum_at, last_purge_at

-- Note: events, metrics, and relations tables are added by
-- Stage 5, Stage 6, and Stage 7 migrations respectively.

-- Indexes
CREATE INDEX idx_memories_key_scope ON memories(key, scope) WHERE deleted_at IS NULL;
CREATE INDEX idx_memories_source ON memories(source_type, source_ref) WHERE deleted_at IS NULL;
CREATE INDEX idx_memories_session ON memories(session_id);
CREATE INDEX idx_memories_hash ON memories(normalized_hash, scope);
```

## Technology Choices

| Choice | Rationale |
|-|-|
| Rust | Cargo workspace: memory-core (lib, WASM-compatible) + memory-agent (bin, native). Single binary (~3MB), ~3-5MB RAM, ~1ms startup. WASM target for universal embedding. |
| WASM compilation | One codebase runs as: native CLI, VS Code extension, browser IDE, edge worker, embedded lib |
| rusqlite | Battle-tested SQLite bindings. FTS5 support built-in. |
| rmcp | Official Rust MCP SDK (github.com/anthropics/model-context-protocol/tree/main/sdk/rust) |
| SQLite + FTS5 | Proven by Engram. Zero infrastructure. FTS5 sufficient for coding memories. |
| MCP-only (no HTTP at L0) | Simplest possible interface. HTTP at L1 if needed. |
| No embeddings at L0 | FTS5 covers keyword search. Removes model dependency from MVP. |
| No TUI at L0 | Skip visual browser. MCP + CLI sufficient. Ratatui TUI is a nice-to-have later. |

6. **Relations** — derived_from, supersedes, conflict detection, cascade (week 6-7)

9. **WASM + Distribution** — cross-compilation, npm/brew, WASM module (week 10-11)

11. **Protocol Spec** — open RFC, conformance tests, agent guides (week 12-13)

## Adoption Strategy

### Phase 1: Wedge
Ship Level 0 as Claude Code plugin + npm package. Replace `.claude/memory/`. Cold start from existing config. Immediate value for existing users.

### Phase 2: Open Standard
Publish protocol spec. Small enough to implement in a weekend. Pitch to Cursor, Windsurf, Cody, Continue, aider: "your users already have memories. Want to read them?"

### Phase 3: Network Effect
User memories portable across agents. The data is the moat, not the implementation.

### Phase 4: Premium Layer
Open protocol + reference implementation. Hosted version with better search (embeddings), cloud sync, team-shared memories, automatic consolidation. Free = local SQLite. Paid = cloud-synced.

## What's NOT In This Architecture

Removed during analysis:
- Separate codebase MCP server (unified into source_type)
- AST-based diff classification (overkill for event processor)
- Intent classifier before retrieval (FTS5 + scope is enough)
- L1/L2/L3 tier system (scope + token budget instead)
- Consolidation model at L0-L1 (simple heuristics; model-driven at L2 only)
- File watcher as primary change signal (git diff primary)
- Graph database (SQLite recursive CTEs suffice)
- Embedding dependency at L0 (FTS5 first, embeddings optional at L2)
- TUI (nice-to-have, not MVP — Ratatui later)
- HTTP server at L0 (MCP-only for simplicity)
- TypeScript/Go (Rust for performance + WASM universality)
- Engram's topic key suggestion heuristics (model is better at classification)
- Engram's passive capture regex (daemon behavioral observation is more robust)
- Git hooks (daemon watches .git directly via notify crate)
- Cloud LLM dependency (Tier 1 zero-LLM is default, Tier 2 local model is opt-in)
- Skill file modification (skills adapt via memory, not by rewriting skill files)
